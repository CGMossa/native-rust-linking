# sys-deps registry

One file per system dependency, recording how the **two Windows toolchains
link it** and where they disagree:

- **Rtools** — what R's build system uses (`pkg-config --libs --static <dep>`
  inside the Rtools45 UCRT64 environment). This is what ends up in an R
  package's `Makevars.win` `PKG_LIBS`.
- **cargo** — what Rust computes for `x86_64-pc-windows-gnu`
  (`--print native-static-libs`, vendored).

## Why

A Rust staticlib linked into an R package is linked by R, not cargo (see
[`../LINKING.md`](../LINKING.md) §2). The two toolchains are not
interchangeable: they differ in C runtime (Rtools45 = UCRT, Rust `-gnu` =
MSVCRT), in where the library comes from, and in build options. They do tend
to agree on the OS libraries the dependency drags in. This registry tracks
exactly where, per dependency.

## Rtools45 reference (provenance)

The Rtools side of every entry depends on the Rtools build, which tracks the R
version. Two facts shape the whole registry:

- **UCRT since Rtools42 (R 4.2).** There is no MSVCRT R anymore (that was
  Rtools40 / R <= 4.1), so the UCRT-vs-MSVCRT mismatch with Rust's
  `x86_64-pc-windows-gnu` applies to every currently-supported R.
- **`pkg-config` arrived in Rtools43**, which is why R's modern build system
  resolves link lines through it — the basis for the "Rtools view" here.

| Rtools | R | GCC | MinGW-w64 | OpenSSL (latest) | notes |
|---|---|---|---|---|---|
| 42 | 4.2 | 10.4.0 | 9.0.0 | 1.1.1q | first UCRT Rtools; zlib 1.2.12 |
| 43 | 4.3 | 12.2.0 | 10.0.0 | 3.1.4 | pkg-config wrapper added (@5863); zlib 1.3.1; OpenSSL 1.1.1 -> 3.x |
| 44 | 4.4 | 13.3.0 | 11.0.1 | 3.4.0 | aarch64 LLVM 17.0.6 |
| 45 | 4.5 | 14.3.0 | 11.0.1 | 3.6.0 | aarch64 LLVM 19.1.7; build 6768 (2026-02-04) |

Current build **6768** (`rtools45-6768-6492.exe`): UCRT, prefix
`x86_64-w64-mingw32.static.posix` (static libs), GCC 14.3.0, MinGW-w64 11.0.1
(12.0.0 deliberately avoided — numerical differences), OpenSSL 3.6.0 (within
rtools45: 3.4.0 @6536 -> 3.5.0 @6691 -> 3.6.0 @6768). Libraries shipped as
`rtools45-toolchain-libs-{base,cross,full}-6768.tar.zst`.

`pkg-config` resolves these link lines, so changelog dependency shifts flow
straight into the registry (e.g. curl gained `nghttp2` @6536 and `secur32`
@6691; pkg-config users get them automatically).

## Format

`<dep>.md` with: the two flag sets (verbatim from CI), a side-by-side table,
and the agree/diverge summary. Raw captures live in [`../results/`](../results)
and are regenerated by [`../.github/workflows/windows-linking.yml`](../.github/workflows/windows-linking.yml).

## Entries

Indexed two ways:

- **By system dependency** (what links it, where the toolchains diverge):
  - [`openssl.md`](openssl.md)
- **By crate** (what your `Cargo.lock` actually pulls in):
  - [`crates.md`](crates.md) — every native-lib / `-sys` crate in the corpus
    mapped to its system dependency, how it resolves it (system, vendored,
    pure-Rust), and whether the vendoring is necessary.
- **Corpus evidence:**
  - [`corpus.md`](corpus.md) — survey of 64 extendr R packages: how often a Rust
    R package needs a provided system library at all (3 of 64), the CRAN vs
    non-CRAN discipline gradient, and a vendoring audit backed by a CI probe.

## Adding one

Run, on a Windows box with Rtools:

```sh
pkg-config --libs --static <dep>                                   # Rtools view
RUSTFLAGS="--print native-static-libs" cargo build \
  --target x86_64-pc-windows-gnu --features openssl-sys/vendored   # cargo view
```

Then write `<dep>.md` following `openssl.md`.
